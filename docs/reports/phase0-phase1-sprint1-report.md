## ⭐ Главное в одном абзаце

> Мы построили не «приложение», а **платформу**, на которой будет работать всё остальное. Пользователь уже может зарегистрироваться, подтвердить почту, включить двухфакторную аутентификацию, увидеть свой первый «рабочий стол» (workspace) и первый бренд. Но **самих AI-агентов и редактора пока нет** — мы сначала вкладывались в фундамент: безопасность, изоляцию данных между клиентами, готовность к мультивалютности и к нескольким языкам, журнал действий, инфраструктуру для будущих агентов и события в реальном времени. Это значит, что когда в Спринте 2 мы начнём подключать Telegram и первого агента, нам не придётся переделывать ничего из основания.

## 1. Что уже умеет приложение глазами пользователя

### 1.1 Регистрация и вход

Пользователь приходит на лендинг (`/`), нажимает «Зарегистрироваться», вводит email + пароль и попадает в личный кабинет. Под капотом:

- При регистрации **автоматически создаётся** его «рабочее пространство» (workspace) и первый «бренд» внутри него — то есть пользователь сразу видит, чем будет управлять.
- На почту приходит **код для подтверждения** — пока его не ввести, часть функций будет недоступна.
- Если пользователь ввёл пароль 10 раз неправильно — на 30 минут ему **блокируется вход** (защита от перебора).

### 1.2 Восстановление пароля

Кнопка «Забыли пароль?» отправляет на почту **одноразовую ссылку**, по которой можно задать новый пароль. Ссылка живёт ограниченное время, после использования сразу гасится.

### 1.3 Двухфакторная аутентификация

В разделе настроек безопасности (`/settings/security`) пользователь может включить **TOTP-аутентификацию** (Google Authenticator / Authy / 1Password). Сразу же выдаются **резервные коды** на случай потери телефона. После включения вход становится двухшаговым: пароль → 6-значный код.

### 1.4 Личный кабинет

После входа пользователь оказывается на `/dashboard`. Сейчас это **простая стартовая страница** — мы ещё не подключили редактор, агентов и календарь публикаций. Но **навигация, тема оформления, мультиязычность и слой данных уже готовы** — добавлять страницы поверх будет быстро.

### 1.5 Языки и формат данных

Интерфейс **сразу мультиязычный** — есть русский (основной) и каркас английского. Никаких «жёстко прибитых» русских строк в коде — все надписи лежат в одном файле-словаре. Это позволит при выходе на международный рынок просто перевести словарь, не переписывая код.

Каждому пользователю в его профиле уже хранится:
- **Язык интерфейса** (`locale` — `ru-RU` по умолчанию)
- **Часовой пояс** (`timezone` — `Europe/Minsk` по умолчанию)
- **Предпочитаемая валюта** (`preferred_currency` — `RUB` по умолчанию)

То есть когда мы дойдём до биллинга и публикаций по расписанию — все нужные настройки у пользователя уже есть.

## 2. Как устроена платформа изнутри (простыми словами)

### 2.1 Архитектура — «модульный монолит»

Мы не делаем сложную микросервисную архитектуру на старте. Всё лежит в **одной большой папке**, но внутри жёстко поделено на модули: «аутентификация», «рабочие пространства», «события», «биллинг» и так далее. Это сильно ускоряет разработку соло-фаундером — но в любой момент можно вынести модуль в отдельный сервис, если он начнёт тормозить.

### 2.2 Изоляция данных между клиентами

Это, наверное, **самое важное** из всего, что сделано в фундаменте. У каждого клиента есть свой `workspace` (рабочее пространство), и внутри него — свои бренды. **На уровне базы данных** мы гарантируем, что **ни один клиент технически не может увидеть данные другого** — даже если случайно ошибёшься в коде запроса.

Это называется **Row-Level Security** (RLS) — Postgres сам фильтрует строки по принадлежности к workspace перед тем, как отдать их приложению. К каждому запросу приложение прикладывает «личность» текущего пользователя, и Postgres дальше не пускает запрос в чужие строки.

Это дороже на старте по разработке, но **на годы вперёд** убирает целый класс багов (когда клиент A видит данные клиента B) — а это самая страшная катастрофа для SaaS.

### 2.3 События в реальном времени

Сейчас, если на сервере что-то поменялось — например, у пользователя изменили роль или нужно перелогиниться — мы **сразу можем уведомить его браузер**. Без перезагрузки страницы, без ручного «обновить F5».

Под капотом это **WebSocket-соединение** между браузером и сервером + Redis-очередь сообщений. Этот механизм понадобится дальше **постоянно**: «AI закончил черновик поста», «модератор пометил пост на проверку», «бюджет на месяц подошёл к 80%», «пора публиковать» — всё это будет лететь по тому же каналу.

### 2.4 Журнал действий (audit)

**Каждое значимое действие** в системе (логин, регистрация, смена пароля, включение 2FA, …) уже сейчас записывается в специальный журнал с указанием **кто, когда, откуда (IP / User-Agent)** это сделал. Этот журнал нужен для:
- Расследования инцидентов («как взломали аккаунт?»)
- Соответствия регуляторным требованиям (GDPR / 152-ФЗ)
- Самого пользователя (видеть свою историю входов в `/settings/security`)

Журнал **разбит на месячные секции** автоматически — это значит, что таблица не вырастет в миллиарды строк за пару лет.

### 2.5 Инфраструктура для будущих AI-агентов

Самих агентов (Content, Publisher, Analyst, …) пока нет, но **слой, на котором они будут работать**, готов:

- **Каталог Skill'ов** — каждая «навыковая» инструкция для агента лежит отдельным `.md`-файлом с метаданными. Это позволит:
  - Включать / выключать конкретные навыки на уровне отдельного бренда
  - Версионировать инструкции (как код)
  - Автоматически проверять перед коммитом, что инструкция не превышает лимит токенов и написана на английском (так дешевле и точнее)
- **Учёт запусков агентов** — заготовлено место в БД, чтобы для каждого запуска агента сохранять, **какие именно навыки** он использовал. Это пригодится для дебаггинга и для биллинга.

### 2.6 Фича-флаги

В систему встроен механизм **выключателей фич** (через self-hosted Unleash). То есть мы можем выкатить новую функцию **только для 10% пользователей** — посмотреть, не падает ли что-то, и потом раскатить на всех. Без передеплоев.

Первый флаг `enable_auto_publish` уже есть в коде — будет рулить тем, может ли AI-агент **сам публиковать пост** или только готовит черновик на ревью.

### 2.7 Защита от повторов

Если пользователь нажал «Опубликовать» два раза подряд (плохой интернет, frustration-click) — **система не создаст два поста**. Защита называется **Idempotency-Key** и работает прозрачно: клиент к каждому запросу прикладывает уникальный «ключ запроса», и сервер запоминает результат на 24 часа. Повторный запрос с тем же ключом возвращает тот же ответ, не дублируя действие.

### 2.8 Наблюдаемость

Уже на этой ранней стадии каждый HTTP-запрос **трейсится** — мы видим, сколько времени он провёл в Postgres, в Redis, в логике приложения. Это OpenTelemetry-стандарт, который дальше будет отправляться в Grafana / Prometheus. Также все ошибки автоматически уходят в Sentry — мы узнаём о них **раньше, чем пользователь успел пожаловаться**.

### 2.9 Заготовка биллинга

Платежей пока нет, но **схема под мультивалютность** уже на месте:
- Тарифные планы (`plans`) — описание, что входит в Solo / Pro / Network
- Цены планов (`plan_prices`) — **разные цены для разных валют** (RUB / BYN / USD)
- Счета (`invoices`) — где-то хранят сумму в валюте клиента и **рядом** — её эквивалент в USD для внутренней отчётности

То есть когда мы дойдём до подключения первой платёжной системы — таблицы, в которые писать, уже есть.

### 2.10 Лимиты «по индивидуальной договорённости»

Помимо стандартных лимитов плана (например, «Pro = до 3 брендов») — у нас есть таблица `tenant_limit_overrides`, в которой можно **для конкретного workspace** подкрутить отдельные потолки: «этому пилоту разрешаем 10 брендов до 30 июня» или «этому пресс-аккаунту — безлимит на месяц по нашей промо-акции». Это нужно для VIP-клиентов, пилотных пользователей и временных промо.

## 3. Безопасность и приватность

| Что | Как защищено |
| --- | --- |
| **Пароли** | Хешируются bcrypt'ом — даже у нас в БД они нечитаемы |
| **Сессии** | JWT короткий (15 минут) + refresh-токен в защищённой cookie (`HttpOnly`, `Secure`, `SameSite=Lax`) на 30 дней |
| **Кража токенов** | Если кто-то попытался использовать **уже использованный** refresh-токен — **вся семья токенов гасится**, пользователю придётся перелогиниться. Это блокирует «угнал refresh — сижу вечно» |
| **Brute-force** | 10 неудачных входов → блок на 30 минут (per email, в Redis) |
| **Двухфакторка** | TOTP (Google Authenticator) + **резервные коды** на случай потери телефона. Секреты в БД зашифрованы поверх (envelope encryption) |
| **Изоляция клиентов** | Row-Level Security в Postgres — невозможно случайно получить чужие строки из приложения |
| **Подбор reset-ссылок** | Одноразовые ссылки с TTL, при использовании сразу гасятся |
| **Утечка ключей** | Все секреты (`SECRET_KEY`, MFA-Fernet-ключи, SMTP-пароль) — только в переменных окружения, в коде их нет |

## 4. Качество и автоматические проверки

Каждый коммит и каждый pull request **автоматически проходит через**:

| Проверка | Что ловит |
| --- | --- |
| **ruff** | Кривое форматирование, неправильные импорты, мёртвый код в Python |
| **biome** | То же самое для TypeScript / JSX |
| **mypy** | Несоответствие типов в Python (`--strict`) |
| **tsc** | То же для TypeScript |
| **pytest** | 219 тестов на backend, в том числе 4 «настоящих» Postgres-теста на изоляцию данных между клиентами |
| **Next build** | Что фронтенд вообще собирается |
| **lint_set_local** | Что мы случайно не написали «утечку» настроек между запросами (`SET app.* WITHOUT LOCAL`) |
| **check_timestamptz** | Что в новой миграции нет «голых» `TIMESTAMP` без часового пояса — иначе на проде в другой зоне получим сдвинутые даты |
| **i18n_audit** | Что русский и английский словари **синхронизированы** + в `.tsx`-компонентах нет жёстко прибитых русских строк |
| **check_system_prompt_lang** | Что инструкции для AI-агентов написаны на английском (так дешевле и точнее) |
| **validate_skills** | Что метаданные навыков агентов валидны и не превышают лимит токенов |

Все эти проверки также запускаются **локально перед каждым коммитом** через `pre-commit` — то есть до сервера никогда не доедет код, который не пройдёт CI.
