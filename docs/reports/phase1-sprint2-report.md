## ⭐ Главное в одном абзаце

> Спринт 1 был про **фундамент** (auth / RLS / events / billing-скелет). Спринт 2 — про **трубу данных**: мы научили платформу **видеть Telegram**. Пользователь может подключить свой канал к бренду через wizard, и сразу же мы умеем (а) пересосать историю последних 500 постов через Bot API, (б) принимать новые посты в реальном времени через webhook, (в) положить их в партиционированную таблицу, (г) поставить им в очередь задачу на построение embedding-вектора для будущего семантического поиска. Параллельно мы добавили **чтение публичных каналов конкурентов** (через user-bot на Pyrogram с зашифрованными сессиями и пулом ротации), **CRUD брендов** с дефолтным брендом и квотами, **дашборд бренда** с превью «5 последних постов канала», и **отдельный личный кабинет** (смена пароля, смена email с подтверждением, список активных сессий). Самих AI-агентов и редактора по-прежнему нет — но **труба, по которой к ним поедут данные, уже работает**.

## 1. Что уже умеет приложение глазами пользователя

### 1.1 Подключение Telegram-канала

В разделе `/dashboard/channels` пользователь видит таблицу подключённых каналов своего активного бренда. Если каналов ещё нет — кнопка «Подключить канал» открывает **wizard из трёх шагов**:

1. Инструкция: добавь нашего бота в админы канала с правом «Публикация сообщений».
2. Введи `@username` канала или его TG-id (для приватных).
3. Платформа сама проверит, что бот действительно админ — и только тогда подключит.

Если бот не админ или у него нет прав на публикацию — пользователь получает **понятное сообщение с подсказкой**, что именно надо поправить в Telegram. Поддерживаются разные форматы ссылок: `@my_channel`, `https://t.me/my_channel`, `https://t.me/c/123456` (для приватных). Канал можно «отвязать» (soft-detach) — сам канал остаётся в общем регистре, но к бренду больше не привязан.

### 1.2 Переключатель брендов

Если у пользователя несколько брендов внутри workspace'а — в шапке появляется **дропдаун с названиями брендов**. Выбор бренда запоминается локально (в браузере) и автоматически прикладывается ко всем последующим запросам через специальный заголовок `X-Active-Brand-Id`. То есть пользователь может управлять несколькими брендами из одного workspace'а без перелогина.

### 1.3 Настройки бренда

На странице `/settings/brands` — **CRUD брендов**: создать новый, переименовать, поменять иконку, **выбрать дефолтный** (тот, который выбирается автоматически после логина). Лимит «сколько брендов разрешено» берётся из тарифного плана (с возможным override для конкретного workspace'а через `tenant_limit_overrides`). Если пользователь упёрся в лимит — получает понятное сообщение «увеличьте тариф или удалите неиспользуемый бренд».

### 1.4 Дашборд бренда

После входа на `/dashboard` пользователь видит **превью последних 5 постов канала** активного бренда (если канал подключён и backfill уже отработал). Это первая «полезная» страница приложения — раньше было пустое место.

### 1.5 Личный кабинет

Появился раздел `/settings/account` с тремя вложенными страницами:

- **Смена пароля** (`/settings/account/password`): ввод текущего пароля + нового + подтверждения. После успеха все остальные сессии пользователя гасятся, и его выкидывает обратно на `/login` через 1.5 секунды. На почту приходит уведомление «ваш пароль был изменён».
- **Смена email с подтверждением** (`/settings/account/email`): двухшаговый процесс. Шаг 1 — ввести текущий пароль и новый email, получить 6-значный код на **новый** адрес. Шаг 2 — ввести код, и адрес переключается. На **старый** адрес приходит уведомление «ваш email был изменён», на случай если новый email — это попытка угона.
- **Список активных сессий** (`/settings/sessions`): таблица всех refresh-сессий с указанием user-agent, IP, когда выдана, когда истекает. Текущая сессия (та, из которой пользователь сейчас сидит) помечена синим бейджем и её нельзя отозвать. Любую другую можно «выкинуть» одной кнопкой — это полезно, если, например, забыл выйти на рабочем компе.

### 1.6 Письма стали красивее

В Спринте 1 письма были только текстовые. Теперь каждое письмо (подтверждение email, восстановление пароля, MFA-предупреждения, смена пароля, смена email) приходит **и в HTML-варианте, и в plain-text**. Почтовый клиент сам выбирает, какой показать. Языки — RU и EN, переключаются по `locale` пользователя.

## 2. Как устроена платформа изнутри (простыми словами)

### 2.1 Общий регистр каналов (Global Channel Registry)

Самое важное архитектурное решение спринта. У нас **две таблицы** для каналов:

- **`channels`** — это **общий регистр**. Один канал = одна запись, без привязки к workspace'у. Если канал `@durov` подключили десять разных клиентов — у нас всё равно **одна** запись на этот канал.
- **`workspace_channels`** — это **привязка** канала к конкретному бренду в конкретном workspace'е. Один и тот же канал может быть привязан к нескольким workspace'ам с разными ролями (`owned` — наш бот админ; `competitor` — публичный, читаем через user-bot).

Зачем так сложно? Чтобы **парсить историю канала один раз**, а не десять. Это критично для дешёвой стоимости платформы — Telegram Bot API имеет жёсткие rate-лимиты, и если каждый клиент будет качать историю одного и того же канала — мы будем упираться в потолок очень быстро.

### 2.2 История канала (history backfill)

Когда канал подключают, мы **в фоне** ставим Celery-задачу: «пересосать последние N постов через Bot API». Все посты складываются в таблицу `channel_posts`, **партиционированную помесячно** через `pg_partman`. Дедуп — по паре `(channel_id, tg_message_id)`, поэтому если backfill запустить дважды — дубликаты не появятся.

Celery — это очередь задач, которая позволяет: (а) не блокировать пользователя на странице — он нажал «Подключить», увидел «канал подключён», и пошёл дальше; (б) равномерно нагружать Bot API, не упираясь в rate-лимиты; (в) повторять задачи с экспоненциальным backoff, если Telegram внезапно ответил «слишком много запросов».

### 2.3 Приём постов в реальном времени (webhook)

Помимо «исторических» постов, мы должны принимать **новые** посты по мере публикации. Для этого `aiogram 3.x` Dispatcher слушает endpoint `/v1/integrations/telegram/webhook`. Когда в канале выходит новый пост — Telegram стучится к нам, мы парсим тело (текст, entities, медиа, реакции, просмотры), сохраняем в ту же таблицу `channel_posts` и **публикуем событие** `channel.post_received` в наш event bus.

Это событие потом подхватит **Celery-задача `embed_channel_post`**, которая дальше зашлёт текст поста в `LLMProvider` для построения embedding-вектора. Сейчас сам пайплайн **не работает на продакшен-нагрузке** — это заготовка под Спринт 3. Но все таблицы, индексы и Celery-инфра уже на месте.

### 2.4 Векторный поиск (embeddings) — инфраструктура

Самая «амбициозная» часть спринта. Мы заранее проложили рельсы под **семантический поиск по постам каналов**:

- Таблица `channel_post_embeddings` — хранит для каждого поста его embedding-вектор (1536-мерный для `text-embedding-3-small`). Тоже **партиционирована помесячно** через `pg_partman`. На каждой партиции — **HNSW-индекс** (специальный индекс для быстрого поиска похожих векторов).
- `pg_cron`-джоба, которая 25-го числа каждого месяца проверяет, что в новой партиции на следующий месяц HNSW-индекс действительно создан (иначе шлёт алёрт в Sentry).
- Абстракция `LLMProvider` с реализациями: `MockLLMProvider` (для CI и dev) и `PolzaLLMProvider` (под Polza.ai — наш будущий провайдер OpenAI-совместимых моделей).
- Celery-задача `embed_channel_post` — берёт пост, шлёт текст в `LLMProvider`, сохраняет вектор в таблицу.

Это **самая дорогая стройка в спринте** — но без неё агенты в Спринте 3 не смогут «искать похожие посты в своей истории» при генерации новых черновиков. Сейчас задача не запускается в продакшене (фича-флагом не включена), но **дёргаешь embed на тестовом канале → embedding появляется в БД → можно делать тестовый поиск через `pgvector`**.

### 2.5 User-bot для публичных каналов (Pyrogram)

Bot API не позволяет читать каналы, в которых **наш бот не админ** — а это все каналы конкурентов клиента. Поэтому помимо `aiogram` (на Bot API) у нас теперь есть **user-bot на `Pyrogram`** — это обычный Telegram-клиент, который ведёт себя как «обычный пользователь, читающий публичные каналы».

Что сделано:

- **Пул аккаунтов** (`telegram_userbot_sessions`) — несколько Telegram-аккаунтов с `api_id` + `api_hash`. Сессии **зашифрованы** в БД через Fernet (envelope encryption), ключи — только в окружении.
- **Ротация** — задачи равномерно распределяются между активными аккаунтами, чтобы ни один не вылетел за rate-лимит и не получил Flood-Wait.
- **Healthcheck** — Celery-задача `userbot_healthcheck` периодически пингует каждый аккаунт; если аккаунт сломан (например, забанили) — он автоматически выводится из пула.
- **API подключения конкурента**: `POST /v1/brands/{id}/competitors` — добавить публичный канал конкурента на чтение (только role=`competitor`, никаких прав на запись). После этого user-bot будет пересосать его историю и подписываться на новые посты.

### 2.6 Бренд-сервис

Раньше «бренд» был просто записью, которая создавалась при регистрации и больше нигде не менялась. Теперь это **полноценный сервис**: создать, переименовать, переключить дефолтный, удалить. На уровне БД — **EXCLUDE-constraint «не более одного дефолтного бренда на workspace»** (это не обычный UNIQUE — это специальный Postgres constraint для частичной уникальности). Квоты («сколько брендов разрешено») — берутся из таблицы `plans` с возможным override.

### 2.7 Тарифный каталог seed

Раньше таблицы `plans` и `plan_prices` были пустые — мы создали схему, но не наполнили её. Теперь Alembic-миграция **засеивает три тарифа**: Solo, Pro, Network — с лимитами и ценами в RUB, BYN, USD. Это означает, что в Спринте 10 (billing v0), когда мы подключим первую платёжную систему — таблицы уже наполнены, и нам останется только настроить webhook платёжного провайдера.

### 2.8 Серверная i18n + HTML-письма

В Спринте 1 i18n была **только на фронте** — словари `ru.json` / `en.json` для UI. Письма были на хардкоженных русских строках. В Спринте 2 мы добавили **серверную i18n через Jinja2-шаблоны**: каждое письмо лежит в `apps/backend/app/locales/{ru,en}/email/` тремя файлами — `.subject.jinja`, `.txt.jinja`, `.html.jinja`. Когда нужно отправить письмо — сервис сам выбирает локаль пользователя и собирает Multipart-сообщение (plain-text + HTML). HTML-шаблоны простые, без CSS-фреймворков — чтобы они одинаково отображались в Gmail, Outlook, Apple Mail и Яндекс.Почте.

### 2.9 Аудит-журнал — новые типы событий

Журнал из Спринта 1 пополнился новыми типами:

- `channel.connected` / `channel.detached` — подключение и отвязка канала
- `channel.history_backfilled` — закончился backfill истории
- `competitor.added` / `competitor.removed` — публичный канал конкурента
- `user.password_changed` / `user.email_changed` — смены чувствительных данных аккаунта (severity = `warning`)
- `session.revoked` — пользователь сам отозвал refresh-сессию

Все события пишутся в ту же партиционированную таблицу `audit_events`, что и в Спринте 1 — расширения схемы не потребовалось.

## 3. Безопасность и приватность

Сверх того, что было в Спринте 1:

| Что | Как защищено |
| --- | --- |
| **Сессии user-bot'ов** | Pyrogram-сессии (содержат `auth_key`) **зашифрованы Fernet** в БД (envelope encryption) — даже если злоумышленник получит дамп БД, без `USERBOT_FERNET_KEY` из окружения он не сможет ничего расшифровать |
| **Привязка бота к каналу** | На каждом подключении канала бот **проверяет свои права через `getChatMember`**: status=`administrator` + `can_post_messages=true`. Если права отозвали — повторный `verify` это покажет |
| **Чужие каналы** | RLS на `workspace_channels` гарантирует: пользователь workspace'а A технически не может увидеть привязки канала к workspace'у B. Сами `channels` без RLS, но **видеть голые записи в общем регистре никому не вредно** — это публичная информация (TG-id, username) |
| **Telegram webhook spoofing** | `POST /v1/integrations/telegram/webhook` проверяет секретный токен в URL (`secret_token` в Bot API) — если кто-то попытается прислать «фейковый» апдейт от Telegram, мы его отбросим |
| **Кража токенов через смену пароля** | После смены пароля — `token_version` пользователя инкрементируется, **все refresh-семьи** одновременно гасятся. Это значит: даже если злоумышленник уже захватил один из refresh-токенов — после смены пароля у него на руках мёртвая бумажка |
| **Угон email** | При смене email — код высылается на **новый** адрес, а уведомление о смене — на **старый**. Это даёт пользователю шанс заметить попытку угона и обратиться в поддержку до завершения переноса |
| **Уникальность email** | Проверка уникальности email делается **дважды**: один раз на стадии запроса (быстрый отказ), второй раз — внутри транзакции confirm (гарантия от race condition между двумя одновременными сменами) |
| **Подбор кодов смены email** | Тот же механизм, что у email verification из Спринта 1: 6-значный код, TTL, ограничение на число попыток ввода, cooldown между resend'ами |
| **Список сессий** | Пользователь видит **только свои** refresh-сессии (фильтр по `user_id`, RLS не нужен). Текущую сессию **запрещено** отзывать через DELETE — иначе пользователь сам себя выкинул бы из приложения без объяснения причин (есть отдельный `/logout` для этого) |

## 4. Качество и автоматические проверки

К проверкам из Спринта 1 (ruff / biome / mypy / tsc / pytest / Next build / i18n_audit / lint_set_local / check_timestamptz / check_system_prompt_lang / validate_skills) добавились:

| Проверка | Что ловит |
| --- | --- |
| **`MockTelegramBotClient` фикстуры** | CI не требует реального `TELEGRAM_BOT_TOKEN` — все тесты на каналы используют заранее заготовленные ответы Bot API. То же и для **`MockUserBotClient`** под Pyrogram |
| **`MockLLMProvider`** | CI не требует ключа Polza.ai — embedding-сервис в тестах возвращает детерминированные «фейковые» векторы фиксированной длины |
| **integration-тесты на `pg_partman` + HNSW** | Отдельная джоба «postgres integration (RLS / PgBouncer parity)» поднимает реальный Postgres с `pgvector` и `pg_partman`, прогоняет миграцию embeddings и проверяет, что **HNSW-индекс действительно создан** на партициях |
| **`i18n_audit` стал строже** | Теперь проверяет **парность по серверным шаблонам писем** — если в RU добавили `subject.jinja`, то и в EN должен быть, иначе билд красный |

Всего тестов на backend стало **существенно больше** — каждая фича (channels-connect, history-backfill, webhook-ingest, embeddings, userbot-pool, brands-CRUD, dashboard, account-settings) покрыта своим test-файлом, плюс сервисные тесты бизнес-логики и API-тесты на ошибки и RLS.
