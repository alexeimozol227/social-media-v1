## 1. Как приоритизируем (MoSCoW)

| Категория | Что значит | Когда делаем |
|---|---|---|
| 🟢 **Must** | Без этого MVP не работает | До запуска MVP |
| 🟡 **Should** | Очень нужно, но MVP можно выпустить и без этого | Сразу после MVP (v1.1) |
| 🔵 **Could** | Хорошо бы иметь | Бэклог (v2+) |
| ⚫ **Won't (пока)** | Сейчас не делаем, чтобы не размывать фокус | Не раньше второго года |

## 2. Что входит в MVP

### 🎯 Критерии «MVP готов»

MVP считается готовым, когда **3 пилотных пользователя** могут:

1. Зарегистрироваться, подключить канал → **Onboarding** извлекает Brand Memory из последних 50 постов **за ≤ 15 минут** (включая режим «холодного старта» для новых каналов)
2. Получить **5 черновиков постов на неделю** по одной кнопке «План на неделю», с поддержкой **MarkdownV2**
3. К каждому черновику получить **AI-сгенерированный визуал**
4. **Moderation** проверил контент перед публикацией (без ручной проверки на токсичность)
5. Опубликовать пост в канал прямо из платформы (вручную или автоматически), координацию выполняет **Orchestrator**
6. Получать **уведомления** от Notification: оперативные («черновик готов», «нужно подтверждение», «модерация пометила») — **только в Telegram-бот** (чтобы не засорять почту); биллинг и еженедельные отчёты — на e-mail
7. Через 7 дней увидеть **базовый отчёт** от Analyst (просмотры, реакции, лучший пост)
8. **Время до первого автономного действия** (TTFAA) **< 2 часов**

### 🟢 MVP Must (P0) — обязательно к запуску

#### EPIC-A: Онбординг и подключение канала (Onboarding + Brand Memory)

| Фича | Описание | Агент |
| --- | --- | --- |
| **A1. Регистрация** | E-mail + пароль; позже — Telegram Login Widget | — |
| **A2. Создание рабочего пространства** | На MVP: 1 пользователь = 1 рабочее пространство. Таблица `Membership` закладывается сразу — чтобы позже подключить команды и агентства без миграций | — |
| **A3. Подключение Telegram-канала** | Пользователь добавляет нашего Telegram-бота админом в свой канал. Платформа проверяет права через Bot API. Каждый канал привязывается к бренду | — |
| **A4. Wizard Brand Memory (стандартный)** | 5–7 вопросов: тематика, аудитория, тон голоса, табу, примеры лучших постов | Onboarding |
| **A5. Автоматическая выгрузка Brand Memory из истории** ⭐ | Onboarding читает последние 50 постов канала через Bot API → извлекает тон, темы, форматы, успешные паттерны → формирует Brand Memory (общий слой + TG-overlay). Пользователь подтверждает или правит | Onboarding + Brand Memory |
| **A6. Расширенный wizard для нового канала** | Анкета на 12–15 вопросов + 5–10 эталонных примеров + 20 предложений тем. Активен, когда у канала меньше 10 постов в истории | Onboarding |
| **A7. Указание референсных каналов конкурентов** | На онбординге пользователь вводит 3–5 публичных каналов в своей нише. Они индексируются через **user-bot с `api_id` + `api_hash` и ротацией аккаунтов** (наш бот в них не админ) и сохраняются в общий реестр каналов | Onboarding |

> **A5 — ключевое изменение:** раньше Brand Memory заполнялась вручную. Теперь Onboarding делает это автоматически — за счёт этого время до первого автономного действия укладывается в 2 часа, а активация — больше 60%.

| Фича | Описание | Агент |
| --- | --- | --- |

| **B5. Brand Memory — учёт стиля** | Каждая генерация использует Brand Memory (общий слой + TG-overlay) + историю канала + референсные каналы. Без этого получится «обычный GPT» | Content + Brand Memory |

| **B7. Генерация в MarkdownV2** | Content выдаёт пост уже размеченным (заголовок жирным, акценты курсивом, спойлеры где уместно) | Content |

#### EPIC-B2: Визуал к посту (Media — базовый) ⭐

| Фича | Описание | Агент |
| --- | --- | --- |
| **B8. Картинка к посту** | Media читает текст поста + визуальный гайдлайн из Brand Memory → формирует промпт → генерация (Flux-2 Pro / Nano Banana 2) → пользователь выбирает или заменяет | Media |
| **B9. Когда нужна картинка** | Media сам решает: новостной дайджест — можно без, how-to — нужна. Пользователь может отключить вообще | Media |
| **B10. Стиль картинки = стиль бренда** | Промпт учитывает Brand Memory: цвета, стиль, тип визуалов (фото / иллюстрация / карточка с цитатой) | Media + Brand Memory |

> **Почему Media сразу в MVP:** стоимость одной картинки умещается в бюджет, картинки увеличивают вовлечение в TG-постах на 30–50%, без них пользователь идёт в Canva, и мы теряем «всё в одном».

Режим «AI делает сам» vs «AI предлагает, человек утверждает»** | Переключатель в настройках бренда + **закрытый каталог готовых правил** (10–15 чекбоксов: макс. длина, без ссылок, тихие часы, лимит в день, evergreen-авто, требовать человека на продающие посты и т.п.). **Свободный текст правил на MVP запрещён** — интерпретация языка слишком ненадёжна для продакшена. #### EPIC-H: Модерация (Moderation) ⭐

| Фича | Описание | Агент |
| --- | --- | --- |
| **H1. Проверка контента перед публикацией** | Каждый пост проходит через Moderation: токсичность, дискриминация, соответствие бренду, табу из Brand Memory | Moderation |
| **H2. Три уровня решений** | Авто-пропустить (если уверен, что ок), пометить на ручную проверку (если сомневается), авто-заблокировать (если уверен, что плохо) | Moderation |
| **H3. Правила + LLM-судья** | Связка: набор простых правил-фильтров (запрещённые слова, регулярные выражения) + LLM-судья (лёгкая модель) для семантической проверки | Moderation |
| **H4. Защита от prompt-инъекций** | Защищает от попыток «обмануть» AI через комменты или темы, которые ввёл пользователь | Moderation |

> **Стоимость:** пренебрежимо мала — копейки за пост (лёгкая модель, маленький запрос).

#### EPIC-F: Платформенные основы

| Фича | Описание |
| --- | --- |
| **F1. Многоарендность с таблицей `Membership`** | Изоляция данных через Postgres RLS. `Membership` как отдельная таблица — чтобы потом подключить команды и агентства без миграций |
| **F2. Аудит решений AI-агентов** | Каждое решение агента логируется: какой промпт, какая модель, какой результат, сколько стоило |
| **F3. Откат публикаций** | Удалить опубликованный пост из канала + из платформы |
| **F5. Настройки** | Профиль, Brand Memory (общий слой + overlay), расписание, режим (auto / approve), список референсных каналов, **вкладка Skills** (вкл/выкл не-системных skills для бренда, превью «когда активируется») |
| **F6. Платформенные роли** ⭐ (D16) | **3 роли на MVP:** `admin` (основатели/инженеры; полный доступ + админ-панель + impersonate), `support` (read-only по аккаунтам + точечные действия: сбросить пароль, переотправить подтверждение, поставить бренд на паузу), `user` (обычный пользователь). MFA обязателен для `admin` и `support`. Роли внутри рабочего пространства (Owner / Editor / Viewer) заложены в схеме, активируются после MVP |
| **F7. Внутренняя телеметрия по стоимости** ⭐ | Каждый вызов LLM помечается `tenant_id` + `workspace_id` + `brand_id` + `agent`. **Видна только в нашей внутренней админ-панели** (для основателя/ops), пользователю не показывается. Для пользователя — простая подписка: оплачиваешь тариф, никакого счётчика расхода в UI |
| **F8. Cost Guardian на каждый бренд** ⭐ | **Технический контроль стоимости LLM (внутренний, пользователю не виден).** Многоуровневые автодействия по средней стоимости поста для бренда: дорого → алерт админу; ещё дороже → автоматический перевод на более лёгкую модель LLM на 24 часа; ещё дороже → **пауза на новые LLM-генерации для бренда на час** (именно генерации, потому что деньги тратит LLM, а не публикация уже готовых постов); критично дорого → **выключение режима «AI делает сам»** (full-auto): бренд переходит в режим «человек утверждает», новые автоматические LLM-цепочки не стартуют, пока админ не разберётся |
| **F8b. Лимиты тарифа подписки** ⭐ | **Лимиты тарифа, которые видит и контролирует сам пользователь, не админ.** Notification уведомляет пользователя при приближении к месячному лимиту тарифа (например, лимит на количество AI-генераций или постов): при 60% — заранее, информативно; при 80% — повторно + предложение перейти на тариф выше; при 100% — **блокируется именно та фича, по которой исчерпан лимит** (например, новые AI-генерации недоступны до апгрейда тарифа или следующего расчётного периода). Уже сгенерированные и запланированные посты продолжают публиковаться по расписанию |
| **F9. Очистка пользовательского ввода для LLM** ⭐ | Всё, что попадает в LLM из публичных каналов / комментариев / Inspiration Board, проходит через очистку: URL → `[URL_REDACTED]`, телефон → `[PHONE_REDACTED]`, bleach + список запрещённых паттернов («ignore previous instructions», zero-width символы, control-символы), допуск только URL из списка наших доверенных доменов |
| **F10. Политика хранения данных** ⭐ | Цепочки рассуждений агентов — 30 дней «горячо», метаданные — 90 дней. Вызовы LLM — 90 дней + дневные агрегаты бессрочно. Сырые посты каналов — 180 дней + 24 месяца в холодном хранилище. Эмбеддинги постов — 12 месяцев активно. Аудит-лог — 2 года «горячо» + 5 лет в холодном. Уведомления — 30 дней. Реализация через `pg_cron`. Обучение наших моделей на данных пользователей — только с явного согласия (opt-in в настройках) |
| **F11. Типизированные ошибки API → понятный UI** ⭐ | Все ошибки агентов / LLM — Pydantic-схемы с `error_code` (`LLM_BUDGET_EXCEEDED`, `MODEL_TIMEOUT`, `CIRCUIT_BREAKER_OPEN`, `MODERATION_BLOCKED`, ...) + `suggested_action` + `retry_after_seconds`. Frontend ловит → понятный toast + CTA («Повторить» / «Экономнее» / «Редактировать вручную» / «Подождать»). **Никаких немых ошибок** — все статусы видны в разделе «Запуски агентов» |
| **F12. Skill-архитектура агентов** ⭐ | Все 8 MVP-агентов используют `SkillCompiler` вместо больших цельных промптов. Skills — переиспользуемые блоки в `app/skills/<name>/SKILL.md` (YAML-шапка + Markdown-тело). На MVP — 5–7 базовых skills (формат TG-MarkdownV2, применение тона бренда, защита от prompt-инъекций, объединение 5 уровней контекста, продающие хуки и CTA, мягкие «вечнозелёные» хуки, оценка авто-правил). Активация — через YAML DSL `when_to_use`. Экономия токенов 28–30%. Версии хранятся автоматически: `agent_runs.skills_used JSONB` с GIN-индексом |
| **F13. Кастомизация skills под бренд** ⭐ | **Уровень 1, на MVP, все тарифы:** в настройках бренда можно выключать не-системные skills. **Уровень 2, на старших тарифах после MVP:** добавлять свои skills (визуальный редактор + Markdown). **Уровень 3, для агентств, после MVP:** переопределять глобальный skill для конкретного бренда. Skills с тегом `safety` или `system` (защита от prompt-инъекций, базовые правила Content) **никогда** не выключаются и не переопределяются |

### ⚫ MVP Won't — точно НЕ в MVP (стоп-лист)

**Соцсети и форматы:**

- ❌ Любая соцсеть кроме Telegram
- ❌ Видео-генерация (включая «текст → короткое видео»)
- ❌ Реактивная аналитика конкурентов в духе TGStat

**Бизнес-модели и инфраструктура:**

- ❌ White-label / режим агентства (в схеме `parent/child workspaces` заложен, но UI не делается)
- ❌ Маркетплейс кастомных агентов
- ❌ On-premise / private cloud
- ❌ Нативные мобильные приложения (iOS/Android)
- ❌ Plugin-система
- ❌ Pay-per-use / счётчик расхода в пользовательском UI — продаём подписку, а не «потребление». Внутренний учёт стоимости остаётся только в админ-панели
- ❌ «Приоритетная поддержка» как платный бонус — поддержка одинаковая для всех тарифов
- ❌ Cloudflare (DNS / CDN / R2 / WAF / Pages) — обходимся без него

**Источники данных:**

- ❌ HTML-парсинг каналов (раньше был обсуждаем; вместо него — Bot API для собственных каналов и user-bot с `api_id` + `api_hash` + ротация для чтения публичных каналов)
- ❌ TDLib
- ❌ Платные API сторонних аналитик (TGStat Pro)

**Защита от рисков:**

- ❌ Агенты для генерации лидов / холодных DM-рассылок (это спам)
- ❌ Клонирование голоса бренда из аудио/видео (далеко за скоупом)
- ❌ Свой дизайн постов (Canva-style шрифты/шаблоны)

**Продуктовые ограничения:**

- ❌ Полная локализация UI на английский (на MVP — только RU)
- ❌ Кастомизация UI / темы оформления

## 4. Нефункциональные требования (NFR) для MVP

| Требование | На MVP | Цель (v1.0+) |
| --- | --- | --- |
| **Многоарендность** | Изоляция через Postgres RLS + middleware | RBAC, командная работа |
| **Производительность** | Генерация поста ≤ 15 секунд; изображения ≤ 10 секунд | ≤ 5 сек текст, ≤ 6 сек изображение |
| **Доступность (uptime)** | По возможности, 95% | SLA 99.5% |
| **Безопасность данных** | HTTPS, шифрование БД, отдельный секрет на аккаунт | SOC2-ready |
| **Удаление / экспорт данных** | Право пользователя на удаление и выгрузку своих данных | — |
| **Логирование AI-решений** | Аудит-лог: модель, версия промпта, цепочка рассуждений, стоимость | UI объяснимости |
| **Резервные копии** | Ежедневные бэкапы БД | Point-in-time recovery, 30 дней |
| **Локализация** | Только RU | RU + EN |
| **Стоимость LLM на пост** | ≤ ~8 ₽ с картинкой; ≤ ~3 ₽ без картинки | ≤ 5 ₽ (кэширование, batch API) |
| **Стоимость LLM на бренд/мес** | ≤ ~250 ₽ при 30 постах/мес | Дополнительная оптимизация |
| **TTFAA** | < 2 часов (медиана) | < 30 минут |
| **Mobile** | Адаптивный веб | PWA → native |
| **Виртуальные списки** ⭐ | Все списки больше 200 строк (лог LLM-вызовов, запуски агентов, доска вдохновения, календарь публикаций) рендерятся через виртуализацию. Без неё UI лагает | Авто-batching, бесконечный скролл |
| **Хранение данных** ⭐ | `pg_cron` чистит и архивирует: цепочки рассуждений — 30 дней; LLM-вызовы — 90 дней + дневные агрегаты; сырые посты — 180 дней + 24 мес в холодном; аудит — 2 года «горячо»; уведомления — 30 дней | Многоуровневое хранение автоматизировано |
| **Очистка пользовательского ввода** ⭐ | Любой внешний текст в LLM-контексте — через `input_sanitizer.py`: URL/телефон редактируются, bleach + список запрещённых паттернов от prompt-инъекций. Покрыто snapshot-тестами | LLM-проверка на всех scrape-входах |
| **Типизированные ошибки** ⭐ | 0 немых ошибок: все ошибки агентов / LLM — Pydantic `AppError` с `error_code`, UI показывает понятный toast + CTA | Авто-восстановление для временных ошибок |
| **Размер JWT** ⭐ | JWT ≤ 2 KB. Только `user_id` + `platform_role` + `active_workspace_id` + `exp` + `jti`. Membership-ы — кэш в Redis (TTL 5 минут) | Поддержка агентств с 500+ пространствами без изменения архитектуры |
| **Партиционирование эмбеддингов** ⭐ | `channel_post_embeddings` разбит по месяцам через pg_partman, шаблон с HNSW. `pg_cron` 25-го числа алертит, если HNSW не появился в новой партиции | Авто-архивация партиций >12 месяцев в S3 |
| **PgBouncer + RLS** ⭐ | Transaction pooling + FastAPI-зависимость `SET LOCAL app.current_tenant_id` в начале каждой транзакции. `SET LOCAL` гасится при commit — соединения возвращаются в пул чистыми. CI запрещает `SET` без `LOCAL` | Read-replica для тяжёлой аналитики |
| **Skill-архитектура** ⭐ | Каждый агент использует `SkillCompiler.compile()` вместо большого цельного промпта. `SkillRegistry` грузит `app/skills/*/SKILL.md` при старте. `agent_runs.skills_used JSONB` + GIN-индекс. Бюджет токенов на skill ≤ 2000, иначе CI падает | Маркетплейс skills для агентств, LLM-судья для выбора skills |
| **Безопасность DSL skills** ⭐ | `when_to_use` парсится через pydantic-yaml (никакого eval). Регулярки — google-re2 (защита от ReDoS). Глубина рекурсии `any_of/all_of/not` ≤ 5. Статический анализ в CI: если skill не активируется ни в одном из ~20 типичных контекстов — CI падает | Визуальный builder skills для агентств |
| **Защита системных skills** ⭐ | Skills с тегом `safety` или `system` **никогда** не отключаются и не переопределяются (харкодед, попытка → 403). Превышение бюджета токенов × 2 в кастомном skill — 400 при сохранении. Тело санитизируется через bleach. Запрещены Jinja-includes и file references | Аудит-лог с diff для каждого override |

> **Примечание по стоимости:** ориентир — около 7–8 ₽ на пост с картинкой и около 3 ₽ без, около 230–250 ₽ в месяц на бренд при 30 постах. Подробный расчёт — в `agent-cost-analysis.md`. Конкретная маржа и юнит-экономика — в `07-monetization.md`, здесь только технический бюджет.

### Внешние зависимости

| # | Зависимость | Риск | Как страхуемся |

### Скоуп-риски

| # | Риск | Как страхуемся |

## 6. Критерии приёмки MVP

MVP можно публично анонсировать, когда:

1. ✅ Все Must-фичи (EPIC A–K) закрыты, все 8 MVP-агентов работают
2. ✅ 3 пилотных пользователя пользуются продуктом ≥ 4 недель
3. ✅ **Процент принятия AI-предложений ≥ 70%**
4. ✅ Стоимость LLM на 1 платящий бренд укладывается в технический бюджет с большим запасом
5. ✅ **Активация ≥ 50%** (% дошедших до первого автономного действия за 24 часа)
6. ✅ **TTFAA < 2 часов** (медиана от регистрации до первого автономного действия)
7. ✅ **Главная метрика продукта ≥ 25** автономных действий на активный бренд в неделю
8. ✅ **Полнота Brand Memory ≥ 80%** к концу первой недели
9. ✅ Lighthouse-аудит платформы ≥ 80
10. ✅ Базовая документация (FAQ, гайд по подключению канала) опубликована
